## Context

当前 PorterClaw Dashboard 使用固定尺寸 900x600px 和无边框窗口（frameless），并在应用内自定义了 macOS 风格的窗口控制按钮（红/黄/绿三色圆点）。这种设计在 Windows 和 Linux 上显得不自然，且限制了用户调整窗口大小的灵活性。

## Goals / Non-Goals

**Goals:**
- 移除自定义窗口控制按钮
- 使用系统原生窗口标题栏（`frame: true`）
- 实现响应式布局，支持任意窗口尺寸
- 保持视觉美观和功能完整

**Non-Goals:**
- 实现窗口拖拽自定义标题栏
- 支持移动端触摸手势
- 添加窗口尺寸记忆功能

## Decisions

### 1. 窗口框架方案
- **决策**: 使用 `frame: true` 替代 `frame: false` + `titleBarStyle: 'hidden'`
- **理由**: 简单可靠，跨平台一致，无需自定义实现
- **替代方案**: 自定义标题栏（rejected - 增加复杂度）

### 2. 响应式布局方案
- **决策**: CSS Flexbox + Grid + Media Queries
- **理由**: 原生支持，无需额外依赖
- **断点设计**:
  - < 600px: 单列布局，堆叠显示
  - 600-900px: 双列布局
  - > 900px: 双列布局，更宽松的间距

### 3. 组件尺寸方案
- **决策**: 使用相对单位（% / flex / grid）替代固定 px
- **理由**: 自动适应容器尺寸
- **保留的最小尺寸**: 状态指示器、按钮等保持固定尺寸以保证可用性

### 4. 导航栏调整
- **决策**: 移除窗口控制按钮区域，导航栏只保留 Logo 和菜单
- **理由**: 窗口控制已交由系统处理
- **菜单居中**: 保持视觉平衡

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 跨平台外观不一致 | 使用 Electron 默认配置，接受各平台原生样式 |
| 小窗口下布局拥挤 | 设置最小窗口尺寸 600x400px |
| 动画性能问题 | 使用 CSS transform，避免重排 |

## Migration Plan

1. 修改 `src/main/index.ts` 窗口配置
2. 移除 `WindowControls` 组件
3. 更新 `Navbar` 组件（移除窗口控制按钮）
4. 重写 CSS 样式（响应式布局）
5. 测试不同窗口尺寸

回滚策略: 恢复原始代码即可。
